PRのURLを渡すだけでJira・Confluenceから背景情報を自動収集してコードレビュー（週6時間削減）

★★★ 設定半日〜 週6時間削減 GitHub MCP Jira MCP Confluence MCP
← 一覧に戻る

概要

GitHub PRのURLを渡すだけで、PR本文・コード差分を取得し、Jiraチケット・Confluenceドキュメントから実装の背景・要件・アーキテクチャ決定事項を自動収集して文脈を把握したコードレビューを実施する。

完成イメージ（出力例）

↓ /review https://github.com/org/repo/pull/123 を実行した際のClaudeの出力例

Claude のコードレビュー出力
## PRレビュー: feat/user-auth-refactor (#123)

【Jiraチケット参照】 AUTH-456「認証フローをJWT→セッション方式に移行」
【Confluenceドキュメント参照】 アーキテクチャ決定記録 ADR-012「セッション管理方針」

---

✅ 要件との整合性
Jira AUTH-456 に記載された「セッション有効期限30分」は session_config.py L.47 で正しく実装されています。
Confluenceの ADR-012 で定めた「Redisをセッションストアとして使用する」方針と一致しています。

⚠️ コード品質・セキュリティ指摘（優先度：高）
1. auth/middleware.py L.83: セッショントークンを URL パラメータに含めています。
   クロスサイトリクエストフォージェリ（CSRF）攻撃のリスクがあります。
   → Cookie（SameSite=Strict）での受け渡しを推奨します。

2. tests/test_auth.py: ログイン失敗時のロックアウト処理のテストが不足しています。
   → 5回連続失敗でのアカウントロックをカバーするテストを追加してください。

💡 コード品質・可読性（優先度：中）
3. utils/session_helper.py L.21–35: 同じ null チェックが3箇所に重複しています。
   → 共通の `validate_session()` 関数に切り出すことを推奨します。

📊 テストカバレッジ
新規追加コード: 68% → 80% 以上を推奨（特に auth/middleware.py が47%）

総評： ロジック全体は ADR に準拠しており方向性は正しいです。
セキュリティ指摘（1番）の修正後にマージ可能と判断します。

設定例（ケース別）

通常のPRレビュー — スキル設定例（review.md）
---
name: review
description: PRのURLを受け取り背景情報を収集してコードレビューを行う
---

1. GitHub MCPでPRの差分・コメント・関連Issueを取得
2. PR本文からJiraチケット番号を抽出してJira MCPで詳細を取得
3. Confluence MCPで関連ドキュメント・ADRを検索
4. 以下の観点でレビューを実施：
   - 要件との整合性（Jiraチケットに記載の受け入れ条件との照合）
   - コードの品質・可読性・テストカバレッジ
   - 既存アーキテクチャとの整合性
5. 指摘事項を優先度（高/中/低）付きで出力
セキュリティ重点レビュー — スキル設定例（security-review.md）
---
name: security-review
description: セキュリティ観点を重点的にコードレビューする
---

1. GitHub MCPでPRの差分を取得
2. 以下のセキュリティ観点を重点的に確認：
   - 認証・認可の漏れ（未認証エンドポイント、権限チェック不備）
   - 入力バリデーション不足（SQLインジェクション、XSS等）
   - 機密情報のハードコード（APIキー、パスワード、トークン）
   - 依存ライブラリの既知脆弱性
   - セッション管理の問題（CSRF、セッション固定化）
3. CWE番号を参照しながら指摘事項を出力
4. 修正例のコードスニペットを提示する
パフォーマンスレビュー — スキル設定例（perf-review.md）
---
name: perf-review
description: パフォーマンス観点を重点的にコードレビューする
---

1. GitHub MCPでPRの差分を取得
2. 以下のパフォーマンス観点を確認：
   - N+1クエリ問題（ORMの遅延ロードによるDBアクセス増大）
   - インデックスの欠落（WHERE/JOIN条件のカラムに対して）
   - 不要なループ・重複処理・巨大なメモリ確保
   - 非同期処理の適切な使用（ブロッキングI/Oの回避）
   - キャッシュ戦略の改善余地
3. 改善前後のオーダー（O(n)→O(1)等）を示して説明する

設定手順

  • 1
    必要なMCPサーバーを追加
    # .claude/settings.json { "mcpServers": { "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"], "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxx" } }, "jira": { "command": "npx", "args": ["-y", "mcp-atlassian"], "env": { "JIRA_URL": "https://your-domain.atlassian.net", "JIRA_EMAIL": "your@email.com", "JIRA_API_TOKEN": "xxx" } } } }
  • 2
    コードレビュースキルを作成
    # .claude/skills/review.md --- name: review description: PRのURLを受け取り背景情報を収集してコードレビューを行う --- 1. GitHub MCPでPRの差分・コメント・関連Issueを取得 2. PR本文からJiraチケット番号を抽出してJira MCPで詳細を取得 3. Confluence MCPで関連ドキュメント・ADR（アーキテクチャ決定記録）を検索 4. 以下の観点でコードレビューを実施： - 要件との整合性 - コードの品質・可読性・テストカバレッジ - パフォーマンス・セキュリティ上の懸念 - 既存アーキテクチャとの整合性 5. レビューコメントをGitHub MCPでPRに投稿（任意）
  • 3
    実行例
    使い方
    /review https://github.com/org/repo/pull/123
  • 4
    チーム用レビューガイドラインを追加（任意）

    review-guidelines.md にチームのコーディング規約・禁止パターン・必須テスト要件を記載すると、ガイドラインに沿ったレビューが実施できる。

確認チェックリスト

  • Jiraチケットの受け入れ条件がすべてコードに反映されているか
  • 既存のユニットテスト・統合テストがすべてパスするか（CIが緑であるか）
  • 新規コードに対応するテストが追加されているか（カバレッジ80%以上を目安に）
  • APIキー・パスワード等の機密情報がコードにハードコードされていないか
  • 本番データベースや外部サービスへの意図しない影響がないか
  • 破壊的な変更（APIの後方互換性の喪失）が含まれている場合、チームに周知されているか
  • AIのレビューコメントをPRに投稿する前に担当者が内容を確認したか

注意点

  • Jira/ConfluenceのAPIトークンはAtlassian管理画面で発行。組織の管理者承認が必要な場合がある。
  • AIのレビューコメントはPRに自動投稿する前に担当者が確認する。誤ったコメントがPRに残ると混乱を招く。
  • セキュリティ上の問題（認証トークンのハードコード・SQLインジェクション等）は自動検出できるが、ビジネスロジックの妥当性判断は人間が必ず行う。
