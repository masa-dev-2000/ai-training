業務モノレポ：経費精算・稼働報告・ブログ執筆・提案資料整理を1リポジトリで管理

★★★★ エンジニア向け 過去のナレッジに即アクセス Git Claude Code GitHub Actions
← 一覧に戻る

概要

経費精算・稼働報告・ブログ記事・提案資料などの業務成果物をひとつのGitリポジトリで管理する。Claude Codeが過去の成果物を参照しながら新しい業務を処理できるため、ナレッジが蓄積するほど出力品質が上がる。バージョン管理・差分確認・ロールバックもGitで行える。

リポジトリ構成

# 業務モノレポの推奨フォルダ構成 business-monorepo/ ├── CLAUDE.md ← Claudeへの全体指示・会社情報 ├── .claude/ │ ├── settings.json ← MCP接続設定 │ └── skills/ ← カスタムスキル │ ├── expense-report.md ← /expense-report │ ├── work-log.md ← /work-log │ ├── blog-draft.md ← /blog-draft │ └── proposal-organize.md ← /proposal-organize ├── expenses/ │ ├── 2026-04/ │ │ ├── receipts/ ← 領収書PDF │ │ └── report.md ← 経費精算書 │ └── templates/ │ └── expense-template.md ├── work-logs/ │ ├── 2026-04-week1.md │ └── ... ├── blog/ │ ├── published/ ← 公開済み記事 │ ├── drafts/ ← 執筆中 │ └── style-guide.md ← 文体ガイド ├── proposals/ │ ├── by-client/ │ │ ├── client-a/ │ │ └── client-b/ │ └── templates/ └── knowledge/ ├── company-profile.md ├── pricing.md └── past-cases.md ← 過去事例（提案参照用）

設定手順

  • 1
    リポジトリを初期化してCLAUDE.mdを作成する
    git init business-monorepo cd business-monorepo
    CLAUDE.md
    ## 業務モノレポ — Claudeへの全体指示

### 会社情報
- 社名：株式会社〇〇
- 事業：〇〇（B2B SaaS）
- 経費精算締め日：毎月25日

### 利用ルール
- 経費精算は expenses/templates/expense-template.md に従う
- ブログ記事は blog/style-guide.md の文体ガイドを必ず参照する
- 提案書は knowledge/past-cases.md の類似事例を参照する
- 個人情報・機密情報を含むファイルは .gitignore に追加すること

### フォルダの使い方
- expenses/YYYY-MM/report.md に当月の経費をまとめる
- work-logs/YYYY-MM-weekN.md に週次稼働を記録する
  • 2
    経費精算スキルを作成する
    .claude/skills/expense-report.md
    ---
name: expense-report
description: 今月の経費精算書を作成する
---

expenses/YYYY-MM/ フォルダの領収書情報と
expenses/templates/expense-template.md を参照して
今月の経費精算書を作成してください。

以下を確認してから生成：
・領収書ファイルの一覧と金額
・勘定科目の分類（交通費・接待費・消耗品等）
・インボイス登録番号の確認状況

出力先：expenses/YYYY-MM/report.md に保存
  • 3
    提案資料整理スキルを作成する
    .claude/skills/proposal-organize.md
    ---
name: proposal-organize
description: 指定クライアント向けの提案資料をまとめて整理する
---

以下の情報をもとに提案資料の目次案と各セクションのドラフトを生成してください。

参照ファイル：
- knowledge/company-profile.md（会社概要）
- knowledge/pricing.md（料金体系）
- knowledge/past-cases.md（類似事例）
- proposals/by-client/[クライアント名]/（過去の資料）

生成する内容：
1. 提案書目次（スライド構成）
2. 課題認識セクション（顧客課題の整理）
3. ソリューションセクション（自社サービスの提案）
4. 導入事例（past-cases.mdから類似案件を引用）
5. 費用・スケジュール

出力先：proposals/by-client/[クライアント名]/draft-YYYY-MM-DD.md
  • 4
    GitHub Actionsで週次レポートを自動生成する
    # .github/workflows/weekly-summary.yml name: 週次業務サマリー生成 on: schedule: - cron: '0 9 * * 1' # 毎週月曜 9:00 JST jobs: generate-summary: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Claude で週次サマリーを生成 run: | claude --headless --print \ "先週（月〜金）の work-logs/ を参照し、週次業務サマリーを生成して work-logs/weekly-summary-$(date +%Y-%m-%d).md に保存してください" env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }} - uses: stefanzweifel/git-auto-commit-action@v5 with: commit_message: "chore: 週次サマリーを自動生成"
  • 5
    機密情報を.gitignoreで保護する
    # .gitignore expenses/*/receipts/*.pdf # 領収書の原本 **/*personal*.md # 個人情報を含むファイル .env # APIキー *.credentials.json

注意点

  • Gitリポジトリにはデフォルトで全ファイルが記録される。領収書・給与データ・個人情報は必ず .gitignore で除外し、ローカル管理またはセキュアなクラウドストレージに保管する。
  • プライベートリポジトリでも、GitHub Copilot等のAIコード補完機能を使う場合は社内データがモデル学習に使われる可能性がある。設定を確認するか、完全ローカルのGitで運用する。
  • 過去データが増えるほどClaude Codeの参照精度が上がるが、コンテキスト上限を超えると参照されなくなる。定期的にアーカイブ（別ブランチや圧縮）する運用を設ける。
  • 複数人で共有するリポジトリの場合、ブランチ戦略とマージルールを最初に決める。各自の作業ブランチから main にPRでマージする流れが安全。
