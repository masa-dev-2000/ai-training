過去記事DBを参照し矛盾チェック → 新記事執筆まで自動化

★★★★ エンジニア向け 大規模ブログの一貫性を担保 ベクトルDB RAG Claude Code
← 一覧に戻る

概要

過去の記事（数百〜数千件）をベクトルDBに格納し、新記事を書く際に関連する過去記事を参照。内容の矛盾・重複・古い情報を検出した上で、既存コンテンツと整合性のとれた新記事を自動生成する。

完成イメージ（出力例）

↓ 新記事「Claude 3.7の使い方」を執筆する際に生成される矛盾検出レポートの例

矛盾検出レポート（Claude の出力）
【新記事の主張】
「Claude 3.7 は拡張思考モードを使うことで推論精度が向上する。無料プランでも利用可能。」

【過去記事との比較】
▼ 記事ID: 20241205-claude-plans（2024/12公開）
  内容：「Claude の拡張思考（Extended Thinking）は Proプラン以上の機能」
  → ★ 矛盾あり：新記事の「無料プランでも利用可能」は過去記事と相反する。
    最新の公式ドキュメントを確認し、どちらが正しいか編集者が判断すること。

▼ 記事ID: 20250315-claude-comparison（2025/03公開）
  内容：「Claude 3.5 Sonnet のトークン上限は 200K」
  → 参考情報：新記事にモデル比較を含める場合は内部リンク推奨。

【重複している内容】
- 記事ID: 20250601-how-to-claude（2025/06公開）に「Claude の基本的な使い方」が詳述されている。
  新記事では重複を避け、上記記事への内部リンクを設置することを推奨。

【更新が必要な古い記事】
- 記事ID: 20231010-ai-tools（2023/10公開）：「拡張思考は実験的機能」と記載。現在は正式リリース済みのため更新が必要。

【参照・引用すべき過去記事】
- /how-to-claude（Claude基本操作）
- /claude-plans（料金・プラン比較）

設定例（ケース別）

矛盾チェックのみ — プロンプト例
新記事のトピック：「〇〇の使い方」
新記事の主張：（下書きまたは主要な主張を貼り付け）

過去記事（ベクトル検索で上位5件を取得）：
（retrieved_docs を貼り付け）

確認してください：
1. 過去記事との内容の矛盾・不整合（あれば記事IDと矛盾箇所を明示）
2. 重複している内容（内部リンクで誘導すべきか）
3. 参照・引用すべき過去記事のURL
矛盾がなければ「問題なし」と出力してください。
新記事生成まで — プロンプト例
新記事を執筆してください。矛盾チェックを行ってから生成すること。 トピック：「〇〇の使い方（2026年最新版）」 対象読者：初心者〜中級者 文字数：2,000〜2,500字 構成：導入→手順→応用→まとめ 過去記事（取得済み）： （retrieved_docs を貼り付け） 出力： 1. 矛盾チェック結果（矛盾があれば警告を先に表示） 2. 矛盾がなければ本文ドラフト 3. 内部リンク候補リスト（過去記事URL）
更新が必要な古い記事の検出 — プロンプト例
以下の最新情報をもとに、更新が必要な過去記事を特定してください。 最新情報：「〇〇は2026年4月にAPIが変更され、旧バージョンは2027年1月に廃止される。」 過去記事データベース（全件のタイトル・公開日・要約）： （article_metadata_list を貼り付け） 出力： 1. 更新が必要な記事リスト（記事ID・タイトル・更新が必要な理由） 2. 優先度（高：即時更新 / 中：次回定期更新時 / 低：必要があれば）

設定手順

  • 1
    記事DBをベクトルDBに格納
    pip install langchain chromadb anthropic # ingest_articles.py（Claude Codeで生成） from langchain.document_loaders import DirectoryLoader from langchain.text_splitter import MarkdownTextSplitter from langchain_community.vectorstores import Chroma loader = DirectoryLoader("./articles/", glob="**/*.md") docs = loader.load() splitter = MarkdownTextSplitter(chunk_size=1000, chunk_overlap=100) chunks = splitter.split_documents(docs) # メタデータ追加：記事タイトル・公開日・カテゴリ vectordb = Chroma.from_documents(chunks, embeddings, persist_directory="./article_db")
  • 2
    矛盾検出ロジックを実装
    矛盾チェックプロンプト
    新しく書こうとしている記事の内容と、関連する過去記事を比較してください。

新記事のトピック：「〇〇の使い方」
関連過去記事（ベクトル検索で取得）：
【記事1（2024/01公開）】：「〇〇はABC方式でXXX...」
【記事2（2025/06公開）】：「〇〇はXYZ方式に変更され...」

確認してください：
1. 過去記事との内容の矛盾・不整合
2. 重複している内容（内部リンクで誘導すべきか）
3. 古くなっている情報（過去記事の更新も必要か）
4. 参照・引用すべき過去記事のURL
  • 3
    一貫性チェック付きの記事生成
    # writer.py（抜粋） def write_article_with_consistency(topic: str, outline: str) -> str: # 関連記事を検索 related = vectordb.similarity_search(topic, k=5) related_content = "\n---\n".join([d.page_content for d in related]) # 矛盾チェック後に記事生成 response = claude.messages.create(model="claude-opus-4-7", messages=[{"role": "user", "content": f"以下の構成で記事を執筆してください。\n" f"過去記事と矛盾しないよう注意し、参照すべき記事には内部リンクを設定してください。\n\n" f"構成：{outline}\n\n関連記事：{related_content}"}]) return response.content[0].text

確認チェックリスト

  • ベクトルDBへの記事インデックスが最新状態になっているか（新記事公開後に更新する運用があるか）
  • ベクトル検索の結果が妥当か（関係のない記事が上位に来ていないか）
  • 矛盾検出レポートを編集者が目視確認しているか（AIの誤判定に注意）
  • 意図的な方針変更による「矛盾」を誤検出として除外する仕組みがあるか
  • 埋め込み（Embedding）の生成コストを把握しているか（大量記事は高額になる場合あり）
  • 内部リンク候補が実際に存在する記事URLを指しているか（404リンクになっていないか）
  • 更新が必要と判定された古い記事の修正フローが決まっているか

注意点

  • ベクトルDBの構築は記事数が多いほど時間がかかる。1,000記事で数十分〜数時間。初回インデックス作成は余裕をもって実施する。
  • AIが「矛盾あり」と判断した場合でも、意図的な方針変更の場合がある。最終判断は編集者が行う。
  • 埋め込み（Embedding）のコストに注意。記事数が多い場合はバッチ処理で一括生成し、差分のみ更新する運用にする。
