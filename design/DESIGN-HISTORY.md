# AI研修 設計履歴・思想記録

> 作成: 2026-04-28  
> 目的: カリキュラム・教材の設計思想、意思決定の経緯、変更の理由を後から参照できるように記録する。

---

## 1. なぜこの研修を作ったのか

### 背景と動機

AIツールが急速に普及する中で、「なんとなく使っている」状態から「業務で使える」状態に引き上げる研修が必要だった。特に中小企業・福祉・地域系の組織では、ITリテラシーにばらつきがあり、「横文字が多くてついていけない」「何が危険で何が安全か分からない」という声が多かった。

### 解決したかったこと

- AIを「なんとなく怖いもの」から「道具として使えるもの」に変える
- 情報漏洩リスクを正しく理解させる（過剰な恐れも無頓着も防ぐ）
- 1回の研修で終わらせず、8回を通して習慣化まで持っていく

### ビジネス的な文脈

外部訓練機関として複数企業に提供し、各企業の助成金（事業展開等リスキリング支援コース）申請をサポートする設計。助成金の要件（出席率・カリキュラム届出・訓練時間）が制約条件となった。

## 2. カリキュラム設計の思想

### 「5つの業務スキル」フレームワーク

最初にカリキュラムを組んだとき、「AIの機能ごと」に教えるか「業務スキルごと」に教えるかを迷った。機能ごと（要約/翻訳/画像生成など）は陳腐化が早く、来年には説明が古くなる。業務スキルごとなら普遍的で、AIが変わっても「伝える/書く/調べる/考える/まとめる」という軸は変わらない。

この考えで v2 カリキュラムを組んだ。

| 軸 | 対応回 |
|---|---|
| 伝える | 第1〜2回 |
| 書く | 第3回 |
| 調べる | 第4〜5回 |
| 考える | 第6回 |
| まとめる | 第7〜8回 |

### 卒業時の姿から逆算する設計

「この研修を受けた人は最終的にどういう状態になっているか？」を先に定義してから1回1回のゴールを設定した。卒業時の姿：

> 日常業務の中でAIを自然に使え、「AIに任せていい仕事」と「自分で判断すべき仕事」を区別できる。情報セキュリティを守りながら、業務効率を実感できている。

この卒業像から、各回のゴールを逆算して設定している。

### 業種対応の設計：3グループ差替方式

1つの教材で全業種に対応しようとすると、お題が抽象的になりすぎる。かといって業種ごとに全教材を作ると運用コストが倍増する。

落とし所として「演習お題だけをJSON外部化し、業種ボタンで切り替える」設計にした。3グループ（A:公共福祉、B:技術専門、C:顧客対応）は実際の受講企業の業種を見て分類した。

## 3. 教材形式の選択

### なぜバニラHTMLか

最初はスライド形式（Google Slides / PowerPoint）を検討した。しかし以下の理由でHTMLを選んだ：

- タイマー機能（全体経過時間 + パート別残り時間）を入れたかった
- 業種切替ボタンをリアルタイムに動かしたかった
- GitHub Pagesでそのままデプロイできる
- CDNやフレームワークへの依存なく、ファイル1つで完結させたい

CDNなし（バニラHTML）にした理由は、インターネット接続が不安定な環境でも動作させるため。また外部ライブラリのバージョン変更で壊れるリスクを避けたかった。

### GitHub Pagesのデプロイ

「Webに上げたいが一番簡単な方法は？」という要求に対してGitHub Pagesを選択。無料・独自ドメイン不要・GitプッシュでCI/CD完結という条件が揃っていた。

## 4. 主要な設計判断

### 横文字禁止方針

「横文字使うとわからないので使わない方針で」というフィードバックを受けて確定。「プロンプト」「リスキリング」等の業界用語は使う場合は必ず日本語の説明を添える。この方針は全教材に一貫して適用する。

### AIの例え：「優秀だけど口が軽い新人」

情報漏洩リスクを説明するときに「AIに何でも話してはいけない」という抽象的な注意では伝わらない。具体的に「優秀だけど口が軽い新人に仕事を頼む感覚」という例えに落とし込んだ。テスター2（経営者）が「これが最も響いた」と評価した。

### OK/NG分類の3段階

「何が危険か」を伝えるときに「禁止/OK」の2択では曖昧なケースに対応できない。「安全に使える（青）/ 注意が必要（黄）/ 入力禁止（赤）」の3段階にした。当初はカラフルな表示にしていたが、「見にくい」というフィードバックを受けてデザインを改修した。

### テーマ：ライトモード（白背景）

最初はダークテーマで実装していた。「全画面で見たい。ライトモードにして」というフィードバックで変更。ライトテーマの方がプロジェクター表示で視認性が高い。

## 5. 変更履歴（主要な改修とその理由）

### 第1回：フォントサイズの段階的拡大

**変更前**: 基本サイズ（小さい）
**変更後**: h1=56px、段落=26px等

**理由**: 「文字がまだまだちっちゃいよ」というフィードバックを複数回受けた。「スライドみたいな感じで、1スライド1メッセージ」という方針と合わせて、大きく余白をとる設計に変更した。

途中で「34pxに統一」しすぎてコンポーネント内のテキストが折り返しだらけになるという問題が起きた。見出しは大きく（56px等）、コンポーネント内は22〜26px等に分けて設定するよう修正した。

### 第1回：コピーボタンのレイアウト修正

**変更前**: absoluteポジションで右上に配置
**変更後**: flexboxで折り返し配置

**理由**: コピーボタンが文字テキストに重なって読めないという問題が発生。absoluteを使わずにflowの中に収めた。

### 第2回：構成の全面再設計

**変更前**: 「5つのテクニックを一覧紹介 → 総合演習」という2部構成
**変更後**: 「1つのお題をStep1〜5で段階的に完成させる」という1本道の構成

**理由**: テスターから「後半に実践が詰まりすぎて何をやっているか分からなくなった」というフィードバックがあった。各ステップが独立したパーツに見え、繋がりが薄かった。「1つのお題を完成させていく」という軸を持たせることで、各テクニックの意味が実感できるようになる。

### 第2回：休憩タイミングの移動

**変更前**: Step1の直後に休憩
**変更後**: Step4・5の直後に休憩

**理由**: Step1は導入の質問ステップで、まだ「頑張った感」がない。実践が山場を越えた後（Step4・5終了後）に休憩を入れた方が、受講者の集中力マネジメントとして自然。

### 第2回：Step1の質問方式の変更

**変更前**: 「1つずつ質問する（会話型）」が初期設定
**変更後**: 「リスト形式でまとめて質問項目を出力してもらう」が初期設定

**理由**: 「いいプロンプトを考えるのがめんどくさい」「音声入力で一気に答えたい」というフィードバックがあった。リスト形式で質問項目をまとめて出してもらい、音声入力でまとめて答えるフローの方が実用的で速い。

### 第2回：振り返りクイズの刷新

**変更前**: オープンクエスチョン3問（自由記述形式）
**変更後**: ○×・選択肢形式の10問

**理由**: 「何か鼻につく。デザインが見にくい」というフィードバック。オープンクエスチョンは自己採点が難しく、受講者が答えを確認できない。○×・選択肢形式にしたほうがシンプルで、正解が確認できて学習効果が高い。

### 第2回：「研究で分かったこと」セクションの削除

**変更前**: 2ページ目に「研究で分かったこと」セクションがあり、研究成果を根拠として提示していた
**変更後**: セクション削除。サブタイトルも「今日の実践で使う型を整理」に変更

**理由**: 「研究で分かったこと というセクションはいらない」というフィードバック。権威付けのための研究紹介は不要で、「実際に使う型」に直結するページにすることで入りをシンプルにした。

## 6. テスト結果と対応

### テスター1（社会福祉協議会職員・初中級）

- **よかった点**: AIにインタビューさせる手法が最も響いた。議事録自動化・Googleフォーム連携への関心が高い。
- **課題**: PCメモリ不足でGeminiが開けない問題が発生 → 事前環境確認チェックリストの必要性を確認。

### テスター2（経営者・設計業・中級）

- **よかった点**: 情報漏洩リスクと学習設定の話が最大の学び。「口が軽い新人」の例えが響いた。
- **課題**: **43分で終了（90分想定の半分以下）** → 中級者向けに発展トピックの追加・時間バッファ設計が必要。AI間比較（ChatGPT/Gemini/Claude/Perplexity）への強い関心も対応が必要。

### 対応方針

- 全回に「中級者向け発展トピック」を追加（Issue #35）
- 各AIの比較情報を第1回か知識ベースに追加（Issue #36）

## 8. 現場フィードバック（企業導入時の課題認識）

> 記録日: 2026-04-28

### 「コンテキストなしのAI利用が組織の方向をバラバラにする」問題

#### 受けたフィードバック

ある企業の経営者から以下の指摘を受けた。

> 「AIが言うことは分かる。でも、コンテキストを踏まえない回答をするので、社員の向いている方向がバラバラになる。社長が考えていることと違うことをしだす可能性がある。現に今もそういう傾向が出ている。」

#### 問題の構造

個人がAIを使う場合、AIが参照するのはその人が入力した情報だけで、**組織の方針・経営者の意図・会社の価値観**は一切入っていない。その結果：

- 社員Aは「お客様第一」で回答を作り、社員Bは「効率優先」で別の回答を作る
- どちらの指示も間違っていないが、会社の方向性とはズレた出力になる
- AIの使用頻度が高まるほど、組織のアウトプットが分散していく

#### 解決の方向性（概念）

**組織のコンテキストをAIに渡す仕組みを作ること**が根本的な解決策になる。具体的には：

- 経営者・管理職の価値観・判断基準をテキスト化してAIの初期指示（カスタム指示）に入れる
- プロンプトひな型に「わが社の方針：○○を最優先する」を最初から組み込む
- チーム共通のプロンプト集（GemやCopilot Agentなど）を整備し、全員が同じコンテキストから出発できるようにする

#### 研修との関係

現行の初級編（全8回）は**個人スキルの底上げ**を目的としており、組織レベルのコンテキスト管理は対象外としている。ただし、以下の点を念頭に置いておく：

- 第2回で教えている「カスタム指示の設定」は、組織コンテキスト共有の入口になりうる
- 中級・上級編を設計する際は、**チームでのAI活用・組織コンテキストの設計**を1テーマとして独立させることを検討する
- 将来的に「経営者・管理職向けAI研修」「組織AI導入支援」を別メニューとして提供できるか検討の余地がある

#### メモ（講師としての心構え）

この問題は研修参加者に伝えるというより、**研修を設計する側が意識しておくべき構造的課題**。個人がうまく使えるようになることと、組織として同じ方向を向くことは別の問題。個人研修の成果が組織の混乱を生むという逆説もあり得る。
